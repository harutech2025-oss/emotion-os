## 2. 대상

- 내부 테스트: 5명
- 외부 파일럿: 20명 이상 (가능하면 30명까지)

내부 테스트 5명을 먼저 진행하여 명확한 오류와 안내 흐름의 문제를 1차로 거른 뒤, 외부 파일럿을 시작한다.

### 참여자 구성 권장

- 가까운 지인: 약 5명 (빠른 피드백 확보)
- 약한 지인 또는 업무 관계자: 약 5명 (호의 편향 완화)
- 추천 또는 모집을 통한 일반 참여자: 10~20명 (실제 사용자 반응 확인)

## 3. 응답 척도

사용자 화면에는 숫자 없는 5단계 언어형 선택지로 표시한다.

```
전혀 그렇지 않다
가끔 그렇다
보통이다
자주 그렇다
매우 그렇다
```

내부 데이터는 0~4 정수로 저장한다.

```
0 = 전혀 그렇지 않다
1 = 가끔 그렇다
2 = 보통이다
3 = 자주 그렇다
4 = 매우 그렇다
```

미응답은 0이 아니라 `null`로 처리한다. 0과 미응답을 명확히 구분하는 것이 분석 단계의 정합성을 보장한다.

## 4. 사전 준비 작업

파일럿 시작일 7일 전까지 다음 작업을 완료한다.

### 4.1 결과 화면 표현 점검

`App.js` 및 관련 UI 문구에서 다음 표현을 점검한다.

- "진단"
- "당신은 ~형입니다"
- "문제 유형"
- "위험 유형"

위 표현은 다음 형식으로 교체한다.

- "현재 ~ 패턴이 감지됩니다"
- "지금은 ~ 반응이 두드러집니다"
- "현재 상태에서 ~ 경향이 높게 나타났습니다"

문구 수정 기준은 `MEASUREMENT_PRINCIPLES.md`를 따른다.

### 4.2 Q4·Q6 문항 보정

`MEASUREMENT_PRINCIPLES.md` 5장(Q6 재정의)과 6장(다른 고위험 차원)의 보정 문항 예시를 따라 정식 문항을 확정한다.

### 4.3 Q5 외부 노출명 점검

`MEASUREMENT_PRINCIPLES.md` 4장의 낙인 위험도 분류에 따라 Q5의 외부 노출 표현이 "우울"이라는 병리적 낙인어를 사용하지 않는지 점검한다.

### 4.4 문서 최신화 확인

- `DATA_SPEC.md` 최신 상태 확인
- `MEASUREMENT_PRINCIPLES.md` 최신 상태 확인
- `PILOT_PROTOCOL.md` 최신 상태 확인

### 4.5 운영 자료 준비

- 참여자 안내문 (비진단 고지 포함)
- 위기 지원 자원 최신 연락처 확인 (확인 출처와 일자 본 문서에 추가 기재)
- 종료 설문 양식
- 위기 표현 감지 시 발송할 안내문 양식
- 데이터 기록 시트 (구글시트 또는 엑셀)
- Day 1 / Day 7 / Day 14 운영 알림 문구
- 오류 제보 채널

## 5. 일정

### Day 0 — 참여 안내

참여자에게 다음 사항을 안내한다.

1. 본 파일럿은 의료 진단이 아니다.
2. 결과는 성격 판정이 아니라 현재 상태 반응 패턴이다.
3. 2주간 3~5회 정도 사용을 권장한다.
4. 가능하면 패치 실행 후 재점검까지 완료한다.
5. 불편할 경우 언제든 중단 가능하다.

이번 파일럿에서 참여자는 평가 대상자가 아니라 자신의 상태 변화를 관찰해 보고하는 운영자(Operator)임을 명시한다.

### Day 1 — 첫 스캔

다음 데이터를 수집한다.

- 스캔 완료 여부
- 소요 시간
- 1순위 Q유형
- 2순위 Q유형
- 가동률
- 결과 설명 이해도
- 추천 패치 클릭 여부

### Day 2~13 — 반복 사용

권장 사용 빈도.

- 주 3회 이상 스캔
- 감정적으로 흔들린 날의 추가 스캔 권장
- 최소 2회 이상 패치 실행
- 최소 2회 이상 재점검

강제 사용은 부과하지 않는다. 자연 사용 가능성을 확인하는 것이 목적이다.

### Day 7 — 중간 재측정

테스트-리테스트 안정성 확인을 위한 중간 재측정을 실시한다. 재측정 직후 다음 두 문항을 추가로 수집한다.

```
1. 지난 1주일간 일상 상태에 영향을 줄 만한 큰 사건이 있었습니까?
   - 예
   - 아니오

2. 지난 일주일간 Stato가 보여준 가동률이 당신의 실제 체감과 어느 정도 일치했나요?
   1~5점 (1: 전혀 일치하지 않음, 5: 매우 일치함)
```

문항 1은 외부 사건 보조 문항이며 `PILOT_PROTOCOL.md` 5장을 따른다.
문항 2는 알고리즘 정확도의 최종 검증이 아니라 결과 납득도를 확인하는 UX-측정 보조 지표로 사용한다.

### Day 14 — 종료 설문

종료 설문 5문항.

```
1. 결과가 내 상태를 어느 정도 설명한다고 느꼈나요?
   1~5점

2. Q유형 이름이나 설명이 불편하게 느껴졌나요?
   예 / 아니오 / 일부

3. 추천 패치를 실제로 해볼 마음이 들었나요?
   1~5점

4. 재점검 과정이 번거로웠나요?
   1~5점

5. 가장 이해가 안 됐거나 불편했던 문장은 무엇인가요?
   자유응답
```

5번 자유응답은 위기 표현 감지 모니터링 대상이다. 위기 표현이 발견되면 `PILOT_PROTOCOL.md` 3.4장 절차를 따른다.

## 7. 1차 파일럿 내부 성공 기준

본 기준은 내부 운영용이며, 외부 노출 시 정성적 표현으로 변환한다 (`PILOT_PROTOCOL.md` 8장 참조).

| 지표 | 기준 |
|---|---|
| 스캔 완료율 | 80% 이상 |
| 결과 이해도 평균 | 3.5/5 이상 |
| 패치 클릭률 | 50% 이상 |
| 재점검 완료율 | 30% 이상 |
| 심각한 낙인·불편 피드백 | 반복 발생 없음 |
| Q4·Q6 출현 | 완전 0이 아니거나, 0이라도 문항 개선 근거 확보 |

표본 크기가 20명 미만으로 진행되는 경우 Q유형 분포 검증은 1차에서 결정 짓지 않고 2차로 이월한다.

## 8. 운영자 시간 예산

운영자(harutech) 예상 투입 시간은 파일럿 기간 중 약 30~40시간으로 잡는다.

이 시간에는 참여자 안내, 중간 알림, 오류 대응, 자유응답 검토, 데이터 정리, 종료 보고서 작성이 포함된다.

파일럿 기간 중 이 시간 예산을 초과하게 만드는 다른 주요 작업은 가능한 한 보류한다.

## 9. 종료 후 산출물

`PILOT_PROTOCOL.md` 7장에 따라 파일럿 종료 후 7일 이내 다음 폴더 구조로 산출물을 작성한다.

```
docs/pilots/2026-05-pilot-1/
├── RESULTS.md
├── IMPROVEMENTS.md
└── NEXT.md
```

산출물 작성 후 본 운영 계획서는 그대로 보존한다. 1차 파일럿이 어떤 계획으로 시작되었는지를 기록으로 남기는 역할을 한다.

## 10. 위기 자원 연락처 확인 기록

파일럿 시작 직전 운영자가 위기 자원을 최신 출처로 확인한 결과를 아래에 기록한다.

```
확인 일자: (파일럿 시작 직전 기재)
확인 출처: (보건복지부 또는 정신건강복지센터 공식 페이지 등)
연락처 목록: (안내문에 반영된 최종 목록)
```

이 항목은 파일럿 시작일에 채워 넣는다. 빈 칸으로 시작하는 것이 아니라, 빈 칸을 채우는 것이 시작 조건이다.
